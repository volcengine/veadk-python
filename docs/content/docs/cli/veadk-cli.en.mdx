## Overview

VeADK provides the following commands to streamline your workflow:

| Command | Description | Notes |
| :-- | :-- | :-- |
| `veadk init` | Generate a project scaffold deployable to VeFaaS | Generates a complete project directory structure including agent definition files, deployment config, and dependency files. Supports both standard agent and web application templates. |
| `veadk create` | Create a new agent in the current directory | Creates the agent project structure, generating a `.env` config file, an `agent.py` agent definition file, and an `__init__.py` package init file. |
| `veadk web` | A frontend debugging UI with short/long-term memory and knowledge base | Starts a local web server, produces a debugging UI URL, configures memory services and knowledge base integration, and can provide SSO sign-in via Agent Identity. |
| `veadk kb` | Knowledge base operations | Creates knowledge base index files, supports multiple backend stores, and generates vectorized documents and retrieval config. |
| `veadk deploy` | Deploy a project to VeFaaS | Generates a deployment package, creates cloud resource config, deploys the agent to Volcengine Function-as-a-Service, and generates an API endpoint. |
| `veadk eval` | Evaluation with different backends | Generates evaluation report files with performance metrics and test results, supporting both local and remote agent evaluation. |
| `veadk prompt` | Optimize the agent system prompt | Generates an optimized prompt file and saves optimization records and version history in the PromptPilot workspace. |
| `veadk uploadevalset` | Eval set operations | Uploads evaluation datasets to the CozeLoop platform, generating eval set records and access permission config on the platform. |
| `veadk pipeline` | Continuous delivery operations | Creates CI/CD config files and sets up an automated build pipeline. |

## Project Initialization

### Introduction

The `veadk init` command initializes a new VeADK project that can be deployed to Volcengine FaaS.

Through an interactive setup flow, this command creates a new VeADK project from a predefined template. It generates a complete project structure with all files, configuration, and deployment scripts required to deploy to the Volcengine cloud platform.

**Available templates:**

- **'template' (default)**: Creates an A2A/MCP/Web server template with a weather-reporter example, suitable for most agent development scenarios.
- **'web_template'**: Creates a web application template with a simple blog example, designed for web apps with UI components.

### Parameters

The `veadk init` command mainly collects project name, description, and other information through interactive prompts, but also accepts the following command-line arguments:

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--vefaas-template-type` | TEXT | Specifies the template type used during project initialization. |
| `--help` | | Show this help message and exit. |

### Examples

Start the interactive initialization flow:

```bash
veadk init
```

After running the command above, you will be guided through the project initialization flow. Follow the prompts to enter the project name, local directory name, Volcengine FaaS application name, Volcengine API Gateway instance name, service name, upstream name, and other details. Using the default project name `weather-reporter` as an example:

![veadk_init](/assets/images/cli/cli_veadk_init.gif)

You can also specify the `web_template` template directly during initialization:

```bash
veadk init --vefaas-template-type web_template
```

After completion, the generated directory structure is as follows:

```bash
weather-reporter/
├────src                    # Agent project source code directory
│   ├────weather_report     # Weather-reporter agent project directory
│   │    ├──────agent.py    # Main agent definition file
│   │    └──────__init__.py # Agent project package init file
│   ├── agent.py            # Main agent definition file
│   ├── app.py              # Main web application definition file
│   ├── requirements.txt    # Project dependency file
│   ├── run.sh              # Project run script
│   └── __init__.py         # Agent project package init file
├────clean.py               # Cleanup script for removing temporary files generated by the project
├────deploy.py              # Deployment script for deploying the project to Volcengine FaaS
└────__init__.py            # Agent project package init file
```

## Creating an Agent

### Introduction

The `veadk create` command creates a new VeADK agent project with template files preset.

This command creates a new agent project directory containing all the files needed to start VeADK agent development, including environment configuration, agent definition, and package init. It handles missing parameters through interactive prompts and performs safety checks on existing directories.

### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `AGENT_NAME` | TEXT | (Optional) The name of the agent to create, which also serves as the directory name. If not provided on the command line, the user is prompted interactively. |
| `--ark-api-key` | TEXT | (Optional) The ARK API key used for model authentication. If not provided, the user is prompted or can configure it later. |
| `--help` | | Show this help message and exit. |

<Callout type="info" title="Note">
  - The agent name serves as both the directory name and the project identifier.
  - The API key can be configured later by editing the `.env` file.
  - The generated agent can be run immediately with the `veadk web` command.
</Callout>

### Examples

Start the interactive creation flow:

```bash
veadk create
```

You can also create an agent by specifying its name directly, for example:

```bash
veadk create location-agent
```

After running the command above, you will be guided through the agent creation flow. First, the system prompts you to choose how to provide the API Key—you can enter it directly or configure it later, as shown below:

![veadk_create](/assets/images/cli/cli_veadk_create.gif)

You can also provide the API Key while creating:

```bash
veadk create location-agent --ark-api-key "xxxxxx"
```

This command creates a `.env` file containing the API Key. You can edit this file later to update the API Key. The full directory structure created by the command is as follows:

```bash
location-agent/
├── .env          # Environment config containing the API key
├── __init__.py   # Python package init file
└── agent.py      # Main agent definition file
```

At this point, you can run the example `location-agent` agent using the `veadk web` command.

## Web Debugging UI

### Introduction

The `veadk web` command starts a local web server so you can interact with and debug your agent in the browser. VeADK is fully compatible with Google ADK's `adk web` command, ensuring seamless integration with the existing Google ADK ecosystem.

This command starts a web server that supports the agent's short-term and long-term memory features. It automatically detects and loads the agent in the current directory and configures the corresponding memory services.

In addition, `veadk web` supports OAuth2/OIDC SSO sign-in via Agent Identity (User Pool). Once enabled, the web debugging UI integrates with Agent Identity's user pool and client config, automatically handling the login redirect and callback.

### Parameters

The `veadk web` command is fully compatible with Google ADK's `adk web` command, supporting the same parameter interface and behavior.

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--port` | INTEGER | Port the web server listens on. Default: 8000. |
| `--host` | TEXT | Host address the web server binds to. Default: 127.0.0.1. |
| `--log-level` | [debug\|info\|warning\|error] | Log output level. Default: info. |
| `--oauth2-user-pool` | TEXT | The Agent Identity User Pool name to enable. |
| `--oauth2-user-pool-client` | TEXT | The Agent Identity User Pool Client name to enable. |
| `--oauth2-redirect-uri` | TEXT | OAuth2 callback URL. Defaults to `http://{host}:{port}/oauth2/callback`. |
| `--help` | | Show this help message and exit. |

**Short/Long-term memory mechanism**:

The VeADK web debugging UI supports the agent's short-term and long-term memory features, which are passed into the web UI as follows:

**Short-term memory passing**:

- The conversation history produced during interaction is automatically saved as short-term memory.
- The web UI associates the agent's short-term memory data via the session ID.
- Each conversation loads the corresponding short-term memory context.

**Long-term memory passing**:

- The agent's knowledge base configuration is passed to the web UI.
- The web UI supports customizing agent behavior based on long-term memory.
- Long-term memory data is loaded automatically via the agent config.

**Memory service integration**:

- Automatically detects and configures the corresponding memory backend service.
- Supports multiple memory storage types (local, mysql, redis, etc.).
- Memory data is synced and updated in real time in the web UI.

### Examples

In your agent project root, run the following command to start the web debugging UI:

```bash
# Start the web server with default config
veadk web
# Specify the port
veadk web --port 8080
```

Enable SSO sign-in via Agent Identity:

```bash
veadk web --oauth2-user-pool my-user-pool --oauth2-user-pool-client my-web-client
```

This command automatically reads the `agent.py` file in the directory where the command runs and loads the `root_agent` global variable from it. After the service starts, it is typically accessible at `http://127.0.0.1:8000`.

An example is shown below:

![veadk_web](/assets/images/cli/cli_veadk_web.gif)

In the selector at the top-left of the UI, you can choose the agent to debug. In this example, you can see the `location-agent` agent you created.

## Knowledge Base

The `veadk kb` command is a command set for managing VeADK knowledge bases.

### `veadk kb add`

Add a file or directory to the specified knowledge base backend.

#### Introduction

This command supports multiple backend types for storing and indexing knowledge content:

- **local**: SQLite-based local file storage, suitable for development and small deployments.
- **opensearch**: An Elasticsearch-compatible search engine, recommended for production environments with large document volumes.
- **viking**: Volcengine's managed vector database service, optimized for semantic search and RAG applications.
- **redis**: An in-memory data store with vector search, suitable for fast retrieval and smaller knowledge bases.

This command can process a single file (such as PDF, TXT, MD, DOCX, etc.) or an entire directory containing multiple supported files.

#### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--backend` | [local\|opensearch\|viking\|redis] | **(Required)** The knowledge base backend type. |
| `--app_name` | TEXT | The application identifier used to organize and isolate knowledge base data. |
| `--index` | TEXT | The knowledge base index identifier, unique within `app_name`. |
| `--path` | TEXT | **(Required)** The file or directory path of the knowledge content to add. |
| `--help` | | Show this help message and exit. |

The `backend` currently supports 4 types: local, opensearch, viking, and redis.

- **local**: Local in-memory knowledge base storage, suitable for development and small deployments.
- **opensearch**: An Elasticsearch-compatible search engine, recommended for production environments with large document volumes.
- **viking**: Volcengine's managed vector database service, optimized for semantic search and RAG applications.
- **redis**: An in-memory data store with vector search, suitable for fast retrieval and smaller knowledge bases.

<Callout type="info" title="Note">
  The `veadk kb add` command needs to convert your document content into vectors. By default, this process uses an Embedding Model service that turns document content into vector representations. You can configure different embedding models as needed, for example Volcengine's Ark large-model platform.

  You can specify the API Key of your embedding model service by setting the `MODEL_EMBEDDING_API_KEY` environment variable in the `.env` file in your project root.

  In addition, if you choose a non-local backend (such as opensearch, viking, redis), you must ensure the relevant environment variables are configured correctly (these can be set in the `.env` file in your project root).
</Callout>

#### Examples

The following example demonstrates adding a single file to a redis-backed knowledge base and using it in subsequent agent calls.

Suppose you have a file named `qa.md` in the current directory with the following content:

````markdown
# Customer Service Knowledge Base

## 1. Company Overview

VE Technology is a high-tech company focused on intelligent customer service and knowledge management. Our product is the **Intelligent Customer Service System**, which provides enterprise customers with efficient, intelligent automated customer service solutions through natural language processing and knowledge base retrieval.

---

## 2. Product Features

- **Automated Q&A**: Quickly responds to common questions based on the knowledge base.
- **Multi-channel access**: Supports web, App, WeChat, Feishu, and other channels.
- **Smart recommendations**: Recommends relevant answers based on context.
- **Data analytics**: Provides user question statistics and customer-service performance reports.
- **Self-service knowledge base management**: Allows non-technical staff to quickly edit knowledge content.

---

## 3. FAQ

### Q1: Which languages does the Intelligent Customer Service System support?

A1: It currently supports **Chinese** and **English**, with Japanese, Korean, and other languages to be added gradually.

### Q2: Can the system integrate with an existing CRM?

A2: Yes. Our system supports seamless integration with mainstream CRM systems (such as Salesforce, Zoho, Kingdee) via API.

### Q3: What happens if the bot cannot answer a user's question?

A3: The system automatically transfers the question to a human agent and logs it on the backend, making it easy for administrators to add it to the knowledge base.

### Q4: How often is the knowledge base content updated?

A4: The knowledge base supports **real-time updates**—changes take effect immediately after an administrator submits them.

---

## 4. Contact Us

- Website: [https://www.example.com](https://www.example.com)
- Support email: support@ve
- Hotline: 400-123-4567
````

You can add it to a local knowledge base with the following command:

```bash
veadk kb add --backend redis --app_name app --path ./qa.md
```

<Callout type="info" title="Note">
  - In this example we use the redis backend. You need to configure environment variables such as `DATABASE_REDIS_HOST`, `DATABASE_REDIS_HOST`, `DATABASE_REDIS_PORT`, `DATABASE_REDIS_PASSWORD` (and the optional `DATABASE_REDIS_USER`) in the `.env` file in your local directory beforehand.
  - You need to confirm that your configured redis service is running normally and that the port matches the one configured in your `.env` file.
  - You need to ensure that the redis service has vector search enabled.
</Callout>

The operation is shown below:

![veadk_kb_add](/assets/images/cli/cli_veadk_kb_add.gif)

Next, you can verify that the knowledge base was added successfully with the following code:

```python
import asyncio

from veadk import Agent, Runner
from veadk.knowledgebase import KnowledgeBase
from veadk.memory import ShortTermMemory

app_name = "app"
user_id = "user"
session_id = "session"

knowledgebase = KnowledgeBase(backend="redis", app_name=app_name, index="v1")

agent = Agent(
    name="customer_service",
    instruction="Answer customer's questions according to your knowledgebase.",
    knowledgebase=knowledgebase,
)
root_agent = agent

runner = Runner(
    agent=agent,
    short_term_memory=ShortTermMemory(),
    app_name=app_name,
    user_id=user_id,
)

response = asyncio.run(
    runner.run(messages="What features does your product have?", session_id=session_id)
)

print(response)
```

<Callout type="info" title="Note">
  - When running the above code, you need to confirm that the redis service is running normally with vector search enabled. This code must also share the same environment configuration as the command above (we recommend running it in the same directory so it can share the same `.env` config file).
  - The `app_name` in the code must match the `app_name` in the command above; in this example it is `app`.
</Callout>

The operation is shown below:

![veadk_kb_test](/assets/images/cli/cli_veadk_kb_test.gif)

## Deployment

### Introduction

The `veadk deploy` command deploys a VeADK project to Volcengine's Function-as-a-Service (FaaS) platform.

This command creates a deployment package from the local project, configures the necessary cloud resources, and manages the entire deployment process, including template generation, file copying, and cloud resource configuration. The deployment flow automatically handles the `requirements.txt` file and, for security reasons, excludes `config.yaml`.

### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--access-key` | TEXT | Volcengine Access Key (AK). If not provided, the `VOLCENGINE_ACCESS_KEY` environment variable is used. |
| `--secret-key` | TEXT | Volcengine Secret Key (SK). If not provided, the `VOLCENGINE_SECRET_KEY` environment variable is used. |
| `--vefaas-app-name` | TEXT | **(Required)** The name of the target Volcengine FaaS application to deploy to. |
| `--veapig-instance-name` | TEXT | (Optional) The Volcengine API Gateway instance name used to configure external API access. |
| `--veapig-service-name` | TEXT | (Optional) The Volcengine API Gateway service name. |
| `--veapig-upstream-name` | TEXT | (Optional) The Volcengine API Gateway upstream name. |
| `--short-term-memory-backend` | [local\|mysql] | The backend type for short-term memory storage. Default: `local`. |
| `--use-adk-web` | | Enable the ADK Web UI for the deployed agent. |
| `--path` | TEXT | The local directory path containing the VeADK project to deploy. Default: the current directory `.`. |
| `--help` | | Show this help message and exit. |

### Examples

From the project root, run the deployment (AK/SK and the FaaS app name are required):

```bash
veadk deploy \
  --vefaas-app-name my-cloud-app \
  --access-key "YOUR_ACCESS_KEY" \
  --secret-key "YOUR_SECRET_KEY"
```

Deploy a project at a specified path and enable the web UI:

```bash
veadk deploy \
  --path ./my-agent-project \
  --vefaas-app-name my-cloud-app \
  --use-adk-web \
  --access-key "YOUR_ACCESS_KEY" \
  --secret-key "YOUR_SECRET_KEY"
```

<Callout type="info" title="Note">
  - You can also set the `VOLCENGINE_ACCESS_KEY` and `VOLCENGINE_SECRET_KEY` environment variables in the `.env` file in your project root to avoid exposing the AK/SK in plaintext on the command line.
  - During deployment you can specify `--veapig-instance-name`, `--veapig-service-name`, and `--veapig-upstream-name` to configure external API access. If not specified, the system uses defaults and creates them automatically. You can use an existing VeAPIG instance and service, or have the system create new ones during deployment.
  - During deployment, you can check the deployment status and logs in the Volcengine [VeFaas Console](https://console.volcengine.com/vefaas/) to confirm a successful deployment.
</Callout>

The detailed deployment logs are shown below:

![Deployment logs](/assets/images/cli/cli_veadk_deploy_vefaas_log.png)

After a successful deployment, you can view your deployed agent application in the Volcengine [VeFaas Console](https://console.volcengine.com/vefaas/).

## Prompt Optimization

### Introduction

The `veadk prompt` command uses Volcengine's PromptPilot service to optimize an agent's system prompt based on feedback and best practices.

This command loads the agent from a specified local file and uses the specified model to intelligently optimize its prompt.

### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--path` | TEXT | Path to the agent file containing the `agent=...` global variable. Default: the current directory `.`. |
| `--feedback` | TEXT | User feedback and suggestions used to optimize the prompt. |
| `--api-key` | TEXT | The API Key for the PromptPilot service. |
| `--workspace-id` | TEXT | The PromptPilot workspace ID. |
| `--model-name` | TEXT | The model name used for prompt optimization. |
| `--help` | | Show this help message and exit. |

### Examples

Taking the optimization of the `weather_reporter` agent's system prompt as an example:

```bash
veadk prompt --path ./weather_reporter/agent.py --feedback "Make the prompt more specific and clear" --api-key "YOUR_API_KEY" --workspace-id "YOUR_WORKSPACE_ID"
```

<Callout type="info" title="Note">
  - You need to first create a project in the Volcengine [PromptPilot Console](https://promptpilot.volcengine.com/) and obtain an API Key and workspace ID.
  - You can also set the `PROMPTPILOT_API_KEY` environment variable in the `.env` file in your project root to avoid exposing the API Key in plaintext on the command line.
  - This command automatically reads the `instruction` content of the `Agent` object in the agent code and uses it as the target for prompt optimization.
</Callout>

In this example, the original `instruction` content is as follows:

```text
Once user ask you weather of a city, you need to provide the weather report for that city by calling `get_city_weather`
```

The result of running this command is as follows:

```text
Optimized prompt for agent weather_reporter:

# Role
You are a weather information provider. When the user asks about the weather of a city, your task is to offer an accurate weather report for that city.

# Task Requirements
- **User Inquiry**: Once the user asks about the weather of a specific city, you must call the `get_city_weather` function to obtain the relevant weather data.
- **Report Content**: The weather report should include at least the current temperature, weather conditions (such as sunny, cloudy, rainy), and wind speed of the city. If available, you can also provide additional information like humidity and air quality.
- **Format**: Present the weather report in a clear and easy - to - understand format. For example: "In [City Name], the current temperature is [Temperature], the weather condition is [Weather Conditions], and the wind speed is [Wind Speed]."

# Additional Notes
- If the user's inquiry is not clear about the city name or contains ambiguous information, ask the user to clarify the city name before proceeding.
- Ensure that the information provided is based on the data obtained from the `get_city_weather` function and is as accurate as possible.
```

## Evaluation

### Introduction

The `veadk eval` command performs a comprehensive evaluation of an agent using a specified evaluation dataset and metrics.

This command supports two evaluation modes:

- **Local evaluation**: Loads and evaluates the agent from local source code.
- **Remote evaluation**: Connects to and evaluates an agent deployed in A2A mode via a URL.

It also supports two evaluation frameworks:

- **adk**: Google's Agent Development Kit evaluation framework, providing standardized metrics.
- **deepeval**: A more advanced evaluation framework supporting customizable metrics including GEval and tool-use accuracy.

### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--agent-dir` | TEXT | (Local evaluation) The local directory path of the agent to evaluate. Must contain an `agent.py` file that exports a `root_agent` variable. Default: the current directory `.`. |
| `--agent-a2a-url` | TEXT | (Remote evaluation) The full URL of a deployed A2A (Agent-to-Agent) mode agent. |
| `--evalset-file` | TEXT | **(Required)** Path to the evaluation dataset file in Google ADK format. |
| `--evaluator` | [adk\|deepeval] | **(Required)** The evaluation framework to use—choose `adk` or `deepeval`. |
| `--judge-model-name` | TEXT | The name of the language model used for evaluation judgment. Default: `doubao-1-5-pro-256k-250115`. This parameter has no effect with the `adk` evaluator. |
| `--volcengine-access-key` | TEXT | The Access Key (AK) used for Volcengine model authentication. |
| `--volcengine-secret-key` | TEXT | The Secret Key (SK) used for Volcengine model authentication. |
| `--help` | | Show this help message and exit. |

<Callout type="info" title="Note">
  - You must provide either `--agent-dir` or `--agent-a2a-url`.
  - If both are provided, `--agent-a2a-url` takes precedence.
  - You need to prepare an evaluation dataset file in advance. Refer to the instructions in [Evaluation](/en/docs/framework/deploy/evaluation) to create a JSON file in Google ADK format.
</Callout>

### Examples

**Local evaluation example**:

```bash
veadk eval \
  --agent-dir ./my-agent \
  --evalset-file ./eval.json \
  --evaluator adk
```

**Remote evaluation example**:

```bash
veadk eval \
  --agent-a2a-url http://my-agent-url.com/invoke \
  --evalset-file ./eval.json \
  --evaluator deepeval \
  --volcengine-access-key "YOUR_AK" \
  --volcengine-secret-key "YOUR_SK"
```

## Eval Set

### Introduction

The `veadk uploadevalset` command uploads evaluation dataset entries from a local JSON file to the CozeLoop platform for agent evaluation and testing.

This command processes evaluation cases in Google ADK format and converts them to the format expected by CozeLoop.

### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--file` | TEXT | **(Required)** Path to the JSON file containing the dataset entries. |
| `--cozeloop-workspace-id` | TEXT | The CozeLoop workspace ID. If not provided, the `OBSERVABILITY_OPENTELEMETRY_COZELOOP_SERVICE_NAME` environment variable is used. |
| `--cozeloop-evalset-id` | TEXT | The CozeLoop eval set ID. If not provided, the `OBSERVABILITY_OPENTELEMETRY_COZELOOP_EVALSET_ID` environment variable is used. |
| `--cozeloop-api-key` | TEXT | The CozeLoop API Key. If not provided, the `OBSERVABILITY_OPENTELEMETRY_COZELOOP_API_KEY` environment variable is used. |
| `--help` | | Show this help message and exit. |

### Examples

```bash
veadk uploadevalset \
  --file ./my_eval_set.json \
  --cozeloop-workspace-id "YOUR_WORKSPACE_ID" \
  --cozeloop-evalset-id "YOUR_EVALSET_ID" \
  --cozeloop-api-key "YOUR_API_KEY"
```

## Continuous Delivery

### Introduction

The `veadk pipeline` command integrates your VeADK project with Volcengine Pipelines for automated CI/CD deployment.

This command sets up a complete CI/CD pipeline. Whenever changes are pushed to the specified GitHub repository, the pipeline automatically builds, containerizes, and deploys your VeADK agent project. It creates all the necessary cloud infrastructure, including Container Registry (CR) resources, FaaS functions, and pipeline configuration.

### Parameters

| Parameter | Type | Description |
| :--- | :--- | :--- |
| `--veadk-version` | TEXT | The base VeADK image tag used for containerization. Can be 'preview', 'latest', or a specific version. |
| `--github-url` | TEXT | **(Required)** The GitHub repository URL of your project. |
| `--github-branch` | TEXT | **(Required)** The GitHub branch of your project. |
| `--github-token` | TEXT | **(Required)** The GitHub token used to manage your project. |
| `--volcengine-access-key` | TEXT | Volcengine Access Key (AK). If not set, the `VOLCENGINE_ACCESS_KEY` environment variable is used. |
| `--volcengine-secret-key` | TEXT | Volcengine Secret Key (SK). If not set, the `VOLCENGINE_SECRET_KEY` environment variable is used. |
| `--region` | TEXT | The region for Volcengine VeFaaS, CR, and Pipeline. Default: `cn-beijing`. |
| `--cr-instance-name` | TEXT | The Volcengine Container Registry (CR) instance name. Default: `veadk-user-instance`. |
| `--cr-namespace-name` | TEXT | The CR namespace name. Default: `veadk-user-namespace`. |
| `--cr-repo-name` | TEXT | The CR repository name. Default: `veadk-user-repo`. |
| `--vefaas-function-id` | TEXT | The Volcengine FaaS function ID. If not set, a new function is created automatically. |
| `--help` | | Show this help message and exit. |

<Callout type="info" title="Note">
  - The GitHub token must have appropriate repository access permissions.
  - All Volcengine resources are created in the specified region.
  - The pipeline is triggered immediately after creation for the initial deployment.
</Callout>

### Examples

```bash
veadk pipeline \
  --github-url https://github.com/your-user/your-repo \
  --github-branch main \
  --github-token YOUR_GITHUB_TOKEN \
  --volcengine-access-key YOUR_AK \
  --volcengine-secret-key YOUR_SK \
  --region cn-beijing
```
