This document introduces the concept of **tracing** in agent systems and its role in development and practice. Tracing is a core component for achieving observability in enterprise-grade agent applications.

## Full-Chain Observability

Tracing refers to the **full-chain recording and tracing** of an agent's execution. In an agent system, this includes:

- The receipt time and content of user input and requests;
- The agent's internal reasoning process, such as model calls, tool execution, and policy decisions;
- The invocation records of each tool, service, or external system interface;
- The generation of responses and the output results.

Through tracing, an agent's execution process can be turned into analyzable, traceable log data, providing a basis for development, debugging, monitoring, and optimization.

**Key characteristics:**

- **Visualized traces**: The complete flow of each request, from intake to response, can be traced;
- **Structured data**: Supports recording context, state, event types, latency, and more;
- **Cross-component tracing**: Covers the agent, tools, memory modules, knowledge bases, and external interfaces.

## Key Capabilities of Tracing

Introducing tracing into an agent system brings value in several ways:

**Debugging and Problem Localization**

- Agents may encounter logic errors, tool-call failures, or abnormal model output during multi-turn interactions or complex tasks;
- Tracing helps developers quickly locate where a problem occurs.

**Performance Analysis and Optimization**

- By tracking the execution time of each module, you can discover performance bottlenecks;
- Optimize model-call strategies, tool execution order, or caching strategies to improve system response speed.

**Auditability and Compliance**

- Enterprise applications need to record operation chains to meet audit or compliance requirements;
- Tracing can provide a complete record of requests, responses, and decision processes.

**Multi-Agent System Coordination**

- In multi-agent collaboration scenarios, tracing helps analyze the invocation relationships and data flow between agents;
- It supports visualized tracing of cross-agent scheduling and tool calls.

**Improved System Reliability**

- Detect abnormal or failure events in time, and trigger alerts or automatic recovery mechanisms;
- Provide stability guarantees for long-running agent systems.

Through tracing, developers can achieve **enhanced observability** for agent systems, enabling efficient development, reliable operation, and continuous optimization.

## Enabling Observability in VeADK

You can enable full-chain observability for Volcengine agent execution in VeADK:

<Tabs items={['Code', 'Required Environment Variables']}>
  <Tab value="Code">
    ```python title="examples/tracing/tracing_exporter.py"
    import asyncio
    from veadk import Agent, Runner
    from veadk.tracing.telemetry.exporters.apmplus_exporter import APMPlusExporter
    from veadk.tracing.telemetry.opentelemetry_tracer import OpentelemetryTracer

    # init tracing exporter
    exporters = [
        APMPlusExporter(),
    ]
    tracer = OpentelemetryTracer(exporters=exporters)

    # Define the Agent
    agent = Agent(
        tracers=[tracer],
    )

    runner = Runner(agent=agent)

    response = asyncio.run(runner.run(messages="hi!"))
    print(response)
    ```
  </Tab>
  <Tab value="Required Environment Variables">
    Environment variable list:

    - `OBSERVABILITY_OPENTELEMETRY_APMPLUS_API_KEY`: The API Key of the APM service
    - `OBSERVABILITY_OPENTELEMETRY_APMPLUS_ENDPOINT`: The endpoint of the APM service, e.g. `http://apmplus-cn-beijing.volces.com:4317`
    - `OBSERVABILITY_OPENTELEMETRY_APMPLUS_SERVICE_NAME`: The service name for APM, e.g. `python_coder_agent`

    Or define them in `config.yaml`:

    ```yaml title="config.yaml"
    observability:
      opentelemetry:
         apmplus:
            endpoint: ...
            api_key: ...
            service_name: ...
    ```
  </Tab>
</Tabs>
