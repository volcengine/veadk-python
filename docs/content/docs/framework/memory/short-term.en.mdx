title: "Short-term Memory"
description: "VeADK short-term memory keeps session-level context so multi-turn interactions stay coherent."
---

Short-term memory keeps "session-level" context so the agent remembers what was said earlier in a conversation. It is essentially the conversation context sent to the model — the system prompt plus message history. VeADK keys it by `session_id`: reuse the same `session_id` and the agent remembers earlier turns.

When a user starts a conversation, `ShortTermMemory` (or the underlying `SessionService`) automatically creates a `Session` object that tracks and manages all content for that conversation.

<Callout type="info">
Short-term memory builds on Google ADK's Session mechanism. For background, see [Google ADK Session](https://google.github.io/adk-docs/sessions/session).
</Callout>

## Using short-term memory

The following creates short-term memory with the `sqlite` backend and runs two turns under the same `session_id` — the second turn can reference the first:

```python
import asyncio

from veadk import Agent, Runner
from veadk.memory.short_term_memory import ShortTermMemory

APP_NAME = "memory_demo"
SESSION_ID = "user-42-chat"


async def main() -> None:
    # `sqlite` persists the session to a local file; use "local" for a purely
    # in-memory session that disappears when the process exits.
    short_term_memory = ShortTermMemory(
        backend="sqlite",
        local_database_path="./short_term_memory.db",
    )

    agent = Agent(
        name="memory_agent",
        instruction="You are a concise assistant. Remember what the user tells you.",
        short_term_memory=short_term_memory,
    )

    runner = Runner(
        agent=agent,
        short_term_memory=short_term_memory,
        app_name=APP_NAME,
    )

    # Turn 1: tell the agent something.
    print(
        "Turn 1 ->",
        await runner.run(
            messages="My name is Xiaoming and my favorite color is blue.",
            session_id=SESSION_ID,
        ),
    )

    # Turn 2: same session_id, so the agent recalls turn 1.
    print(
        "Turn 2 ->",
        await runner.run(
            messages="What is my name? What color do I like?",
            session_id=SESSION_ID,
        ),
    )


if __name__ == "__main__":
    asyncio.run(main())
```

<Callout type="info">
The model API key comes from the `MODEL_AGENT_API_KEY` environment variable by default — you don't need to put it in code. See [Configuration](/en/docs/framework/configuration).
</Callout>

## Short-term memory backends

In VeADK you can initialize short-term memory with the following backends:

| Backend | Description |
| :- | :- |
| `local` | In-memory short-term memory, cleared when the process exits. Good for local debugging. |
| `sqlite` | Persists to a local SQLite file; the session survives process restarts. Good for single-host persistence. |
| `mysql` | Stores in a MySQL database for distributed persistence. |
| `postgresql` | Stores in a PostgreSQL database for distributed persistence. |

<Callout type="info">
Both `local` and `sqlite` require no external service; the difference is that `local` lives only in memory while `sqlite` is written to `local_database_path`. For distributed production architectures, use `mysql` or `postgresql`.
</Callout>

## Session management

You usually do not create or manage `Session` objects directly. Instead, they are managed through `SessionService`, which handles the full session lifecycle. Its core responsibilities include:

1. **Start a new session** `create_session()`: creates a new `Session` when the user begins interacting.
2. **Resume an existing session** `get_session()`: retrieves a specific `Session` by `session_id`, letting the agent continue from prior progress.
3. **Save conversation progress** `append_event()`: appends new interaction content (`Event` objects) to the session history. This is also how session state is updated.
4. **List sessions** `list_sessions()`: queries active sessions for a given user and application.
5. **Clean up sessions** `delete_session()`: deletes the `Session` and its associated data when the session ends or is no longer needed.

## Context compaction

As a conversation runs, its history keeps growing, which increases the data the model processes and slows responses. Context compaction uses a sliding window to summarize history: when the history exceeds a threshold, older events are compacted automatically.

### Configuring context compaction

Once configured, the `Runner` automatically compacts the session history whenever the interval is reached.

```python
from google.adk.apps.app import App, EventsCompactionConfig

from veadk import Agent

root_agent = Agent(
    name="my_agent",
    instruction="You are an intelligent assistant who replies politely in Chinese.",
)

app = App(
    name="my_agent",
    root_agent=root_agent,
    events_compaction_config=EventsCompactionConfig(
        compaction_interval=3,  # trigger compaction every 3 new invocations
        overlap_size=1,         # overlap with the last event of the previous window
    ),
)
```

### Defining a compactor

Use `LlmEventSummarizer` to specify the model and prompt template used for compaction. The model's API key / base URL come from environment variables — read them from `os.environ` rather than hardcoding:

```python
import os

from google.adk.apps.app import App, EventsCompactionConfig
from google.adk.apps.llm_event_summarizer import LlmEventSummarizer
from google.adk.models.lite_llm import LiteLlm

from veadk import Agent

root_agent = Agent(
    name="my_agent",
    instruction="You are an intelligent assistant who replies politely in Chinese.",
)

# Custom model used by the compactor
summarization_llm = LiteLlm(
    model="volcengine/doubao-seed-1-8-251228",
    api_key=os.environ["MODEL_AGENT_API_KEY"],
    api_base=os.environ.get(
        "MODEL_AGENT_API_BASE", "https://ark.cn-beijing.volces.com/api/v3/"
    ),
)

my_compactor = LlmEventSummarizer(
    llm=summarization_llm,
    # Custom compaction structure
    prompt_template="""Please summarize the conversation. Compression requirements:
1. Retain key entities, data points, and timelines;
2. Highlight core problems and solutions discussed;
3. Maintain logical coherence and contextual relevance;
4. Eliminate duplicate expressions and redundant details.""",
)

app = App(
    name="my_agent",
    root_agent=root_agent,
    events_compaction_config=EventsCompactionConfig(
        compactor=my_compactor,
        compaction_interval=5,
        overlap_size=1,
    ),
)
```
