---
title: "Long-term Memory"
description: "VeADK long-term memory persists important information across sessions for personalized, sustainable agent interactions."
---

Long-term memory persists important information across sessions and over time — user preferences, task history, key knowledge points, or long-term state. While [short-term memory](/en/docs/framework/memory/short-term) only lives within a single conversation, long-term memory lets the agent remember facts across **different** sessions.

Why it matters:

- Supports continuous conversational experiences across sessions;
- Lets the agent retain learning outcomes and user-specific information across many interactions;
- Reduces repeated questions, improving satisfaction and efficiency;
- Supports long-term strategy optimization, such as personalized recommendations or task tracking.

## Supported backends

Use the `local` backend for debugging. For production, the `viking` or `mem0` backends are recommended.

| Backend | Description |
| :----------- | :--------------------------------------------------------- |
| `local` | In-memory cross-session memory, cleared when the process exits (local debugging only). Requires `veadk-python[extensions]`. |
| `viking` | Uses VikingDB as storage (recommended for production). |
| `mem0` | Uses Mem0 as storage (recommended for production). |
| `opensearch` | Uses OpenSearch as storage, with persistence and retrieval. |
| `redis` | Uses Redis as storage; Redis must support RediSearch. |
| `viking_mem` | Deprecated; automatically converted to `viking` when set. |

## Initialization

Before using long-term memory, instantiate `LongTermMemory`, specify the backend, and bind it to an `Agent`. Passing `long_term_memory` to the `Agent` automatically gives it a `load_memory` tool that can search past sessions at runtime.

```python
from veadk import Agent, Runner
from veadk.memory.long_term_memory import LongTermMemory

APP_NAME = "ltm_demo"

# backend="viking" uses VikingDB; app_name is used for data isolation
long_term_memory = LongTermMemory(backend="viking", app_name=APP_NAME)

root_agent = Agent(
    name="ltm_agent",
    instruction="Answer the user's question. Use the `load_memory` tool if the answer might be in past conversations.",
    long_term_memory=long_term_memory,
)

runner = Runner(agent=root_agent, app_name=APP_NAME)
```

<Callout type="info">
Backends like `local`, `viking`, and `mem0` embed memories, so they require the extensions dependency: `pip install "veadk-python[extensions]"`, plus an embedding model config (it reuses `MODEL_AGENT_API_KEY` when omitted).
</Callout>

## Memory management

### Adding a session to long-term memory

When a session ends or reaches a milestone, explicitly call `add_session_to_memory` to persist the session data. It is an async method, so you must `await` it. For the Viking backend, this step triggers vectorization.

```python
# Assume runner has completed a conversation
completed_session = await runner.session_service.get_session(
    app_name=APP_NAME, user_id=USER_ID, session_id=SESSION_ID
)

# Archive the full session to long-term memory
await long_term_memory.add_session_to_memory(completed_session)
```

### Retrieving long-term memory

Besides the agent retrieving memory automatically at runtime via `load_memory`, you can call `search_memory` directly for semantic search — useful for debugging or building custom RAG applications. This method is async and takes keyword arguments:

```python
response = await long_term_memory.search_memory(
    app_name=APP_NAME,
    user_id=USER_ID,
    query="favorite project",
)
print(response.memories)
```

## Cross-session example

The following is a complete, runnable end-to-end flow: tell the agent a fact in session #1, enable `auto_save_session=True` so VeADK archives the session automatically, then ask a question in a **brand-new** session #2 — the agent recalls the fact by searching long-term memory (not its context window).

This uses the `local` backend, which requires `pip install "veadk-python[extensions]"`.

```python
import asyncio

from veadk import Agent, Runner
from veadk.memory.long_term_memory import LongTermMemory

APP_NAME = "ltm_demo"
USER_ID = "user-42"


def build_runner() -> Runner:
    long_term_memory = LongTermMemory(backend="local", app_name=APP_NAME)
    agent = Agent(
        name="ltm_agent",
        instruction=(
            "You are a personal assistant. When the user asks about something "
            "they told you before, use the `load_memory` tool to recall it."
        ),
        long_term_memory=long_term_memory,
        auto_save_session=True,  # persist each finished session to long-term memory
    )
    return Runner(agent=agent, app_name=APP_NAME, user_id=USER_ID)


async def main() -> None:
    runner = build_runner()

    # Session #1: share a fact. auto_save_session stores it afterwards.
    print(
        "Session 1 ->",
        await runner.run(
            messages="Note this down: I'm allergic to peanuts and I'm vegetarian.",
            session_id="session-1",
        ),
    )

    # Session #2: a *different* session. The agent must recall via memory search.
    print(
        "Session 2 ->",
        await runner.run(
            messages="Recommend a dish for me, keeping my dietary restrictions in mind.",
            session_id="session-2",
        ),
    )


if __name__ == "__main__":
    asyncio.run(main())
```

In session #2 the agent recognizes the preferences the same user left in session #1 and gives a coherent, personalized answer (for example, recommending a peanut-free vegetarian dish).

## Automatically saving sessions

As shown above, enabling `auto_save_session=True` when initializing the `Agent` (with long-term memory configured) makes VeADK save sessions to long-term memory automatically, without manually calling `add_session_to_memory`.

```python
from veadk import Agent
from veadk.memory.long_term_memory import LongTermMemory

long_term_memory = LongTermMemory(backend="viking", app_name="ltm_demo")

agent = Agent(
    name="ltm_agent",
    auto_save_session=True,
    long_term_memory=long_term_memory,
)
```

To avoid frequent index initialization, VeADK provides two environment variables, `MIN_MESSAGES_THRESHOLD` and `MIN_TIME_THRESHOLD`, to customize the save cycle. By default a save is triggered after 10 accumulated events or 60 seconds. In addition, when you switch `session_id` and start a new exchange, VeADK automatically saves the previous session's content to long-term memory.
