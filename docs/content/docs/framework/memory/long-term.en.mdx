This document introduces the concept and usage of **Long-term Memory** in VeADK. Long-term memory persists important information across sessions and over time, enhancing the consistency and intelligence of an agent across continuous interactions.

**Definition**: Long-term memory is the mechanism an agent uses to store important information that extends beyond a single session. It can include user preferences, task history, key knowledge points, or long-term state.

**Why it matters**:

- Supports continuous conversational experiences across sessions;
- Allows the agent to retain learning outcomes and user-specific information across many interactions;
- Reduces repeated questions, improving user satisfaction and efficiency;
- Supports long-term strategy optimization, such as personalized recommendations or task tracking.

Agent users need long-term memory to achieve smarter, more personalized, and sustainable interactions, which significantly improve system usefulness in multi-session or complex-task scenarios.

## Supported Backends

You can use the `local` backend for debugging. For production, the `viking` or `mem0` backends are recommended.

| Type | Description |
| :----------- | :--------------------------------------------------------- |
| `local` | In-memory cross-session memory, cleared when the program exits (for local debugging only). |
| `viking` | Uses the VikingDB memory product as long-term memory storage (recommended for production). |
| `mem0` | Uses the Mem0 memory product as long-term memory storage (recommended for production). |
| `viking_mem` | Deprecated; automatically converted to `viking` when set. |
| `opensearch` | Uses OpenSearch as long-term memory storage, with persistence and retrieval. |
| `redis` | Uses Redis as long-term memory storage; Redis must support the Redisearch feature. |

## Initialization

Before using long-term memory, you need to instantiate a `LongTermMemory` object and specify the backend type. The following code shows how to initialize a VikingDB-based long-term memory module and bind it to an Agent:

<Tabs items={['Python', 'Golang']}>
  <Tab value="Python">

```python
from veadk import Agent, Runner
from veadk.memory import LongTermMemory

# Initialize long-term memory
# backend="viking" specifies the use of VikingDB
# app_name and user_id are used for data isolation
long_term_memory = LongTermMemory(
    backend="viking", app_name="local_memory_demo", user_id="demo_user"
)

# Bind long-term memory to the Agent
root_agent = Agent(
    name="minimal_agent",
    instruction="Acknowledge user input and maintain simple conversation.",
    long_term_memory=long_term_memory,  # long-term memory instance
)

runner = Runner(agent=root_agent)
```

## Memory Management

### Adding a Session to Long-term Memory

When a session ends or reaches a specific milestone, you need to explicitly call `add_session_to_memory` to persist the session data. For the Viking backend, this step triggers vectorization of the data.

<Tabs items={['Python', 'Golang']}>
  <Tab value="Python">

```python
# Assume runner1 has completed a conversation
completed_session = await runner.session_service.get_session(
    app_name=APP_NAME, user_id=USER_ID, session_id=session_id
)

# Archive the full session to long-term memory
root_agent.long_term_memory.add_session_to_memory(completed_session)
```

### Retrieving Long-term Memory

In addition to the Agent retrieving memory automatically at runtime, developers can also call the `search_memory` API directly for semantic search, which is useful for debugging or building custom RAG (Retrieval-Augmented Generation) applications.

<Tabs items={['Python', 'Golang']}>
  <Tab value="Python">

```python
query = "favorite project"
res = await root_agent.long_term_memory.search_memory(
    app_name=APP_NAME,
    user_id=USER_ID,
    query=query
)

# Print the retrieval results
print(res)
```

## Managing Sessions with Long-term Memory

In single-tenant scenarios, long-term memory can be used to manage multiple sessions for the same user, ensuring the agent can:

- Remember the previous interaction in a new session;
- Make personalized responses based on historical information;
- Accumulate progress information or intermediate results across multi-turn tasks.

### Preparation

- Assign a unique identifier to each user (`user_id` or `session_owner_id`);
- Design the long-term memory data structure to support saving information across multiple sessions;
- Use it together with short-term memory to enable fast access to in-session context.

### Example

The following example demonstrates a complete flow: Runner1 tells the Agent a piece of information ("My favorite project is Project Alpha"), saves the session to memory, then creates a brand-new Runner2 to verify whether it can answer related questions.

<Tabs items={['Python', 'Golang']}>
  <Tab value="Python">

```python
# --- Phase 1: Write to memory ---
# Runner1 tells the Agent the information
runner1_question = "My favorite project is Project Alpha."
user_input = types.Content(role="user", parts=[types.Part(text=runner1_question)])

async for event in runner1.run_async(user_id=USER_ID, session_id=session_id, new_message=user_input):
    pass # Handle Runner1's response

# Key step: archive the session to VikingDB
completed_session = await runner1.session_service.get_session(app_name=APP_NAME, user_id=USER_ID, session_id=session_id)
root_agent.long_term_memory.add_session_to_memory(completed_session)

# --- Phase 2: Cross-session read ---
# Initialize Runner2 (simulating a new session or follow-up interaction)
runner2 = Runner(
    agent=root_agent,
    app_name=APP_NAME,
    user_id=USER_ID,
    short_term_memory=short_term_memory
)

# Runner2 asks a question, relying on long-term memory to answer
qa_question = "favorite project"
qa_content = types.Content(role="user", parts=[types.Part(text=qa_question)])

final_text = None
async for event in runner2.run_async(user_id=USER_ID, session_id=session_id, new_message=qa_content):
    if event.is_final_response():
         final_text = event.content.parts[0].text.strip()
```

```go
    if err != nil {
        log.Printf("NewLLMAgent failed: %v", err)
        return

    cfg := &veagent.Config{
        ModelName:    common.DEFAULT_MODEL_AGENT_NAME,
        ModelAPIBase: common.DEFAULT_MODEL_AGENT_API_BASE,
        ModelAPIKey:  utils.GetEnvWithDefault(common.MODEL_AGENT_API_KEY),
    }
    cfg.Name = "MemoryRecallAgent"
    cfg.Instruction = "Answer the user's question. Use the 'search_past_conversations' tools if the answer might be in past conversations."

    cfg.Tools = []tool.Tool{memorySearchTool}

    memorySearchAgent, err := veagent.New(cfg)
    if err != nil {
        log.Printf("NewLLMAgent failed: %v", err)
        return
    }

    // Use all default config
    //sessionService, err := vem.NewShortTermMemoryService(vem.BackendShortTermPostgreSQL, nil)
    //if err != nil {
    //	log.Printf("NewShortTermMemoryService failed: %v", err)
    //	return
    //}
    sessionService := session.InMemoryService()
    memoryService, err := vem.NewLongTermMemoryService(vem.BackendLongTermViking, nil)
    if err != nil {
        log.Printf("NewLongTermMemoryService failed: %v", err)
        return

    runner1, err := runner.New(runner.Config{
        AppName:        appName,
        Agent:          infoCaptureAgent,
        SessionService: sessionService,
        MemoryService:  memoryService,
    })
    if err != nil {
        log.Fatal(err)
    }

    s, err := sessionService.Create(ctx, &session.CreateRequest{
        AppName:   appName,
        UserID:    userID,
        SessionID: SessionID,
    })
    if err != nil {
        log.Fatalf("sessionService.Create error: %v", err)
    }

    s.Session.State()

    userInput1 := genai.NewContentFromText("My favorite project is Project Alpha.", "user")
    var finalResponseText string
    for event, err := range runner1.Run(ctx, userID, SessionID, userInput1, agent.RunConfig{}) {
        if err != nil {
            log.Printf("Agent 1 Error: %v", err)
            continue

        if event.Content != nil && !event.LLMResponse.Partial {
            finalResponseText = strings.Join(textParts(event.LLMResponse.Content), "")
        }
    }
    log.Printf("Agent 1 Response: %s\n", finalResponseText)

    // Add the completed session to the Memory Service
    log.Println("\n--- Adding Session 1 to Memory ---")
    resp, err := sessionService.Get(ctx, &session.GetRequest{AppName: s.Session.AppName(), UserID: s.Session.UserID(), SessionID: s.Session.ID()})
    if err != nil {
        log.Fatalf("Failed to get completed session: %v", err)
    }
    if err := memoryService.AddSession(ctx, resp.Session); err != nil {
        log.Fatalf("Failed to add session to memory: %v", err)
    }
    log.Println("Session added to memory.")

    log.Println("\n--- Turn 2: Recalling Information ---")

    runner2, err := runner.New(runner.Config{
        AppName:        appName,
        Agent:          memorySearchAgent,
        SessionService: sessionService,
        MemoryService:  memoryService,
    })
    if err != nil {
        log.Fatal(err)
    }

    s, _ = sessionService.Create(ctx, &session.CreateRequest{
        AppName:   appName,
        UserID:    userID,
        SessionID: "session2222",
    })

    userInput2 := genai.NewContentFromText("What is my favorite project?", "user")

    var finalResponseText2 []string
    for event, err := range runner2.Run(ctx, s.Session.UserID(), s.Session.ID(), userInput2, agent.RunConfig{}) {
        if err != nil {
            log.Printf("Agent 2 Error: %v", err)
            continue
        }
        if event.Content != nil && !event.LLMResponse.Partial {
            for _, part := range event.Content.Parts {
                finalResponseText2 = append(finalResponseText2, part.Text)
            }
        }
    }
    log.Printf("Agent 2 Response: %s\n", strings.Join(finalResponseText2, ""))

}

func textParts(Content *genai.Content) []string {
    var texts []string
    for _, part := range Content.Parts {
        texts = append(texts, part.Text)
    }
    return texts
}
```

## Automatically Saving Sessions to Long-term Memory

To simplify the workflow, the VeADK Agent module supports automatically saving conversation Sessions to long-term memory. You only need to enable the `auto_save_session` attribute when initializing the Agent and complete the long-term memory component initialization. An example is shown below:

```python
from veadk import Agent
from veadk.memory import LongTermMemory

# Initialize the long-term memory component
long_term_memory = LongTermMemory(
    backend="viking", app_name=APP_NAME, user_id=USER_ID
)

# Initialize the Agent and enable automatic session saving
agent = Agent(
    auto_save_session=True,
    long_term_memory=long_term_memory
)
```

To avoid frequent index initialization, VeADK provides two environment variables, `MIN_MESSAGES_THRESHOLD` and `MIN_TIME_THRESHOLD`, which let you customize the session save cycle. By default, a save is triggered when 10 events have accumulated or 60 seconds have elapsed. In addition, when you switch sessions and initiate a new question-and-answer request, VeADK automatically saves the previous session's content to long-term memory.
