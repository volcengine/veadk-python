Defining an agent is the starting point of its lifecycle. This page shows how to configure an agent — choose a model, set up fallbacks, customize the model client, and describe an agent declaratively in YAML.

<Callout type="info">
  Just want to run a minimal example first? Start with the [Quickstart](/en/docs/framework/getting-started/quickstart), then come back here to configure further.
</Callout>

VeADK offers two ways to define an agent: the **code approach** and the **YAML approach**. Code is more flexible — easy to adjust parameters dynamically and unit-test. YAML is more concise and suited to declarative configuration management.

## Build in code

Defining an agent directly in code is the most common and flexible approach. Default attributes come from environment variables or built-in defaults, so the simplest agent needs no extra setup:

```python title="agent.py"
import asyncio

from veadk import Agent, Runner

agent = Agent()
runner = Runner(agent=agent)

response = asyncio.run(runner.run(messages="Hi!"))
print(response)
```

Before running, provide your model inference API key via an environment variable (VeADK reads `MODEL_AGENT_API_KEY` by default):

```bash
export MODEL_AGENT_API_KEY="<your Ark API key>"
```

<Callout type="info">
  You can also manage secrets centrally in `config.yaml`; see [Configuration](/en/docs/framework/configuration). **Never hard-code an API key in source.**
</Callout>

### Set metadata

Set an agent's metadata with `name`, `description`, and `instruction`:

```python title="agent.py"
from veadk import Agent

root_agent = Agent(
    name="life_assistant",
    description="Life assistant",
    instruction="You are a life assistant who can answer user questions.",
)
```

- `name`: the agent's name; uniquely identifies an agent within the agent tree.
- `description`: a short description of what the agent does (surfaced in A2A and similar scenarios).
- `instruction`: the system prompt that defines the agent's behavior and response style.

### Use a different model

To use another provider or a local model, specify the model configuration at initialization (the API key still comes from the environment variable):

```python title="agent.py"
from veadk import Agent

agent = Agent(
    model_provider="openai",
    model_name="doubao-seed-1-8-251228",
    model_api_base="https://ark.cn-beijing.volces.com/api/v3/",
)
```

The inference model in the VeADK Python version is built on [LiteLLM](https://github.com/BerriAI/litellm), so you can use any provider LiteLLM supports. See the [list of providers supported by LiteLLM](https://docs.litellm.ai/docs/providers) to set `model_provider`.

### Model fallbacks

VeADK provides model fallbacks based on LiteLLM [Fallbacks](https://docs.litellm.ai/docs/completion/reliable_completions): when the target model still fails after multiple retries, it automatically switches to a backup model to continue. Pass a list to `model_name`:

```python title="agent.py"
from veadk import Agent

agent = Agent(
    model_name=[
        "doubao-seed-1-8-251228",
        "deepseek-r1-250528",
    ]
)
```

VeADK tries the first model first; if it fails, it tries the subsequent models in order until one succeeds.

### Customize your model client

For highly customized model configuration, pass a LiteLLM client directly:

```python title="agent.py"
from google.adk.models.lite_llm import LiteLlm
from veadk import Agent

# Customize your model client
llm = LiteLlm(model="doubao-seed-1-8-251228")

# Pass the model client directly to the Agent
agent = Agent(model=llm)
```

### Configure extra model parameters

You can also specify [extra options](https://www.volcengine.com/docs/82379/1494384?lang=zh) based on the capabilities of the [Volcengine Ark model platform](https://www.volcengine.com/product/ark). For example, disable the thinking capability for faster responses:

```python title="agent.py"
import asyncio

from veadk import Agent, Runner

agent = Agent(
    model_name="doubao-seed-1-8-251228",
    model_extra_config={"extra_body": {"thinking": {"type": "disabled"}}},
)
runner = Runner(agent=agent)

response = asyncio.run(runner.run(messages="Hi!"))
print(response)
```

## Attach tools, memory, and a knowledge base

Of an agent's [four core elements](/en/docs/framework/agent/overview), the model is configured above; the other three are attached at initialization.

### Tools

Put tool functions or tool instances in the `tools` list. VeADK ships built-in tools, and you can also pass plain Python functions as custom tools:

```python title="agent.py"
from veadk import Agent
from veadk.tools.demo_tools import get_city_weather

agent = Agent(
    name="weather_assistant",
    instruction="Call tools to answer the user's weather questions.",
    tools=[get_city_weather],
)
```

See [Built-in tools](/en/docs/framework/tools/builtin) and [Custom function tools](/en/docs/framework/tools/custom-function).

### Memory

Pass `ShortTermMemory` (single-session context) to the `Runner`, and `LongTermMemory` (cross-session context) to the `Agent`. When long-term memory is attached, VeADK automatically injects the `load_memory` tool so the model can retrieve history on demand:

```python title="agent.py"
from veadk import Agent, Runner
from veadk.memory.short_term_memory import ShortTermMemory
from veadk.memory.long_term_memory import LongTermMemory

agent = Agent(
    name="assistant",
    long_term_memory=LongTermMemory(backend="local"),
)

runner = Runner(agent=agent, short_term_memory=ShortTermMemory(backend="local"))
```

See [Short-term memory](/en/docs/framework/memory/short-term) and [Long-term memory](/en/docs/framework/memory/long-term).

### Knowledge base

Pass a `KnowledgeBase` to `knowledgebase`. Once attached, VeADK automatically injects the `load_knowledgebase` tool so the model can retrieve relevant snippets before answering:

```python title="agent.py"
from veadk import Agent
from veadk.knowledgebase import KnowledgeBase

agent = Agent(
    name="kb_assistant",
    instruction="Retrieve from the knowledge base before answering.",
    knowledgebase=KnowledgeBase(backend="local"),
)
```

See [Knowledge base](/en/docs/framework/knowledgebase/overview).

## Through YAML

When you want more concise, declarative configuration management, describe an agent's full metadata and behavioral structure in a YAML file. The `weather_reporter_agent.yaml` below defines an agent that queries the weather:

```yaml title="weather_reporter_agent.yaml"
root_agent:
  type: Agent # Agent | SequentialAgent | LoopAgent | ParallelAgent
  name: weather_reporter
  description: An intelligent assistant that can provide weather information.
  instruction: Help the user according to your tools.
  model_name: doubao-seed-1-8-251228
  tools:
    - name: veadk.tools.demo_tools.get_city_weather # The module and function name where the tool lives
```

Field reference:

| Field | Description |
| :--- | :--- |
| `type` | Agent category (`Agent`, `SequentialAgent`, `LoopAgent`, `ParallelAgent`) |
| `name` | Agent name |
| `description` | Overall description of the agent |
| `instruction` | System prompt |
| `model_name` | Model name, such as `doubao-seed-1-8-251228`, `deepseek-r1-250528`, etc. |
| `tools` | Tool list and invocation rules |

<Callout type="info">
  The API key is omitted from YAML and supplied by the `MODEL_AGENT_API_KEY` environment variable.
</Callout>

Load the YAML with `AgentBuilder` and run it:

```python title="agent.py"
import asyncio

from veadk import Runner
from veadk.agent_builder import AgentBuilder

agent_builder = AgentBuilder()
agent = agent_builder.build(path="weather_reporter_agent.yaml")

runner = Runner(agent)
response = asyncio.run(runner.run("What is the weather in Beijing today?"))
print(response)
```

## Comparison

| Feature | Code approach | YAML approach |
| :--- | :--- | :--- |
| Flexibility | High | Medium |
| Readability | Medium | High |
| Maintainability | Medium | High |
| Dynamic generation | Supported | Limited |
| Use cases | Development, production | Experimentation, configuration-driven, production |
