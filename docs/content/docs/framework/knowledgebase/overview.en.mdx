The KnowledgeBase is an agent's external source of knowledge — a "library" of static material it can consult at any time. Attach it to an agent and VeADK automatically gives the agent a retrieval tool, so it can look up relevant passages from your content before answering, producing more precise, grounded responses (RAG).

## Key features

1. **Ingestion**: add external, unstructured text (product docs, FAQs, articles) into the knowledge base; the framework converts it into a retrievable format automatically.
   - From files: `kb.add_from_files([...])`
   - From a directory: `kb.add_from_directory("./docs")`
   - From in-memory text: `kb.add_from_text([...])`
2. **Backend abstraction**: a unified interface hides the underlying vector store. Specify `local`, `viking`, `opensearch`, or `redis` via `backend` at init — your business code stays the same.
3. **Retrieval**: once a knowledge base is passed to an agent, the agent gains a built-in `load_knowledgebase` tool and decides on its own whether to use it when answering.
4. **Seamless agent integration**: pass `knowledgebase=kb` when creating an `Agent`.

## Usage

Here is the minimal flow for using a knowledge base.

<Steps>

<Step>
### Initialize the knowledge base and ingest knowledge

The `local` backend requires no external service, making it the easiest to start with. It embeds documents, so install the extensions dependency `pip install "veadk-python[extensions]"` and configure an embedding model (it reuses `MODEL_AGENT_API_KEY` when omitted).

```python
from veadk.knowledgebase import KnowledgeBase

kb = KnowledgeBase(backend="local", index="company_faq")

# Ingest from in-memory text; you can also use add_from_files / add_from_directory
kb.add_from_text(
    [
        "Standard annual leave is 15 days per year, available after one year of employment.",
        "Employees may work remotely up to 2 days per week with manager approval.",
    ]
)
```
</Step>

<Step>
### Integrate with an agent and run

```python
import asyncio

from veadk import Agent, Runner

agent = Agent(
    name="kb_agent",
    model_name="doubao-seed-1-8-251228",
    instruction="You are a knowledgeable assistant. Prefer the knowledge base when answering.",
    knowledgebase=kb,  # automatically gains the load_knowledgebase tool
)

runner = Runner(agent=agent, app_name="company_faq")

print(asyncio.run(runner.run(messages="How many days of annual leave are there? Can I work remotely?")))
```
</Step>

</Steps>

Switching to a persistent production backend (`viking`, `opensearch`, `redis`) barely changes your code — just change `backend` and provide the connection details in `config.yaml`. The sections below cover provisioning and configuring those two backends.

### Code configuration

#### config.yaml

Fill the provisioned VikingDB, TOS, and OpenSearch details into `config.yaml`. Do **not** put secrets in the file — reference them via environment variables:

```yaml
model:
  agent:
    provider: openai
    name: doubao-seed-1-8-251228
    api_base: https://ark.cn-beijing.volces.com/api/v3/
    api_key: ${MODEL_AGENT_API_KEY}

volcengine:
  # Viking DB and embedding require Volcengine ak/sk
  access_key: ${VOLCENGINE_ACCESS_KEY}
  secret_key: ${VOLCENGINE_SECRET_KEY}

database:
  viking:
    project: default    # project in Volcengine Viking DB
    region: cn-beijing
  tos:
    endpoint: tos-cn-beijing.volces.com
    region: cn-beijing
    bucket: your_bucket_name
  opensearch:
    host: your_opensearch_host
    port: 9200
    username: admin
    password: ${DATABASE_OPENSEARCH_PASSWORD}
```

Provide those secrets in environment variables:

```bash
export MODEL_AGENT_API_KEY="<your Ark API key>"
export VOLCENGINE_ACCESS_KEY="<your AK>"
export VOLCENGINE_SECRET_KEY="<your SK>"
export DATABASE_OPENSEARCH_PASSWORD="<your OpenSearch password>"
```

### Using VikingDB as storage

VikingDB requires explicitly creating a collection once before use. Here we ingest knowledge from in-memory text:

```python
from veadk.knowledgebase import KnowledgeBase

APP_NAME = "viking_demo"

kb = KnowledgeBase(backend="viking", index=APP_NAME)

if not kb.collection_status()["existed"]:
    kb.create_collection()  # Viking requires an explicit create

kb.add_from_text(mock_data)
```

Complete runnable example (VikingDB backend with a date-difference tool):

```python
import asyncio
from datetime import datetime

from veadk import Agent, Runner
from veadk.knowledgebase import KnowledgeBase

APP_NAME = "viking_demo"

mock_data = [
    "Sigmund Freud (6 May 1856 - 23 Sep 1939) founded psychoanalysis.",
    "Alfred Adler (7 Feb 1870 - 28 May 1937) founded individual psychology.",
]

kb = KnowledgeBase(backend="viking", index=APP_NAME)
if not kb.collection_status()["existed"]:
    kb.create_collection()
kb.add_from_text(mock_data)


def calculate_date_difference(date1: str, date2: str) -> int:
    """Calculate the absolute number of days between two dates.

    Args:
        date1: the first date, in "YYYY-MM-DD" format.
        date2: the second date, in "YYYY-MM-DD" format.

    Returns:
        The number of days between the two dates.
    """
    d1 = datetime.strptime(date1, "%Y-%m-%d")
    d2 = datetime.strptime(date2, "%Y-%m-%d")
    return abs((d2 - d1).days)


agent = Agent(
    name="chat_agent",
    model_name="doubao-seed-1-8-251228",
    instruction=(
        "You are an excellent assistant. Query the knowledge base to verify or "
        "enrich your answer, then combine it with your own knowledge for a "
        "complete, accurate response."
    ),
    knowledgebase=kb,
    tools=[calculate_date_difference],
)

runner = Runner(agent=agent, app_name=APP_NAME)

if __name__ == "__main__":
    print(asyncio.run(runner.run(messages="How many days apart are Freud and Adler?")))
```

### Using OpenSearch as storage

OpenSearch usually creates the index automatically, with no explicit create. Here we ingest knowledge from a file and also attach the real `web_search` tool to supplement from the web when the knowledge base is insufficient:

```python
import asyncio
from datetime import datetime

from veadk import Agent, Runner
from veadk.knowledgebase import KnowledgeBase
from veadk.tools.builtin_tools.web_search import web_search

APP_NAME = "opensearch_demo"

kb = KnowledgeBase(backend="opensearch", index=APP_NAME)
kb.add_from_files(["tmp/demo.txt"])  # write your knowledge into tmp/demo.txt first


def calculate_date_difference(date1: str, date2: str) -> int:
    """Calculate the absolute number of days between two dates.

    Args:
        date1: the first date, in "YYYY-MM-DD" format.
        date2: the second date, in "YYYY-MM-DD" format.

    Returns:
        The number of days between the two dates.
    """
    d1 = datetime.strptime(date1, "%Y-%m-%d")
    d2 = datetime.strptime(date2, "%Y-%m-%d")
    return abs((d2 - d1).days)


agent = Agent(
    name="chat_agent",
    model_name="doubao-seed-1-8-251228",
    instruction=(
        "You are an excellent assistant. Query the knowledge base first; "
        "if it is insufficient or the question involves real-time information, "
        "use `web_search`; then combine everything into a complete, accurate answer."
    ),
    knowledgebase=kb,
    tools=[calculate_date_difference, web_search],
)

runner = Runner(agent=agent, app_name=APP_NAME)

if __name__ == "__main__":
    print(asyncio.run(runner.run(messages="How many days apart are Freud and Adler?")))
```

## Summary and comparison

The KnowledgeBase design lets you switch the underlying vector store without changing most of your business logic. The main differences are the upfront configuration and the fact that Viking requires explicitly creating a collection.

| Feature | Viking backend | OpenSearch backend |
| :--- | :--- | :--- |
| **Core advantage** | Managed service, no ops overhead, tightly integrated with the Volcengine ecosystem. | Open-source and flexible, supports private deployment, with a mature community. |
| **config.yaml** | Requires `viking.project`, `viking.region`, and `tos`. | Requires `opensearch.host`, `port`, `username`, `password`. |
| **Initialization** | Check and call `kb.create_collection()` before use. | Usually creates the index automatically, with no extra steps. |
| **Code usage** | **Identical**: `add_from_files`, `add_from_text`, etc. are used the same way. | **Identical**: the abstraction layer hides the differences. |
| **Dependencies** | Relies on Volcengine `ak/sk` for authentication and embedding. | Also requires `ak/sk` to call the embedding model, but the database itself is independent. |
