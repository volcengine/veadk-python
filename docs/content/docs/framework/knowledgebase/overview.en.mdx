## Key Features

In the veadk framework, the KnowledgeBase acts as the Agent's external knowledge base. It is like a "library" of static material that the Agent can consult at any time. Its core features include:

1. **Ingestion**: You can add external, unstructured text material (such as product docs, FAQs, articles, knowledge snippets, etc.) into the knowledge base. The framework automatically processes this text and converts it into a format the Agent can understand and retrieve.
   - Supports importing from files (`kb.add_from_files([...])`).
   - Supports importing from in-memory text strings (`kb.add_from_text([...])`).

2. **Backend Abstraction**: The KnowledgeBase provides a unified interface that hides the implementation details of the underlying vector database. You only need to specify `viking` or `opensearch` via the `backend` parameter at initialization, without worrying about their respective API calls.

3. **Retrieval**: Once a KnowledgeBase instance is passed to an Agent, the Agent automatically gains a built-in `knowledgebase_search` tool. When answering questions, the Agent can autonomously decide whether to use this tool, turning the user's question into keywords to search the knowledge base for relevant information, producing more precise and context-aware answers.

4. **Seamless Agent Integration**: By passing the `knowledgebase=kb` parameter when creating an Agent, the Agent can automatically use this knowledge base to enhance its answers.

## Usage

The following shows the typical steps for using the KnowledgeBase and compares the configuration differences between `viking` and `opensearch`.

**Step 1: Configure the config.yaml file**

This is the main difference between the two. You need to provide the connection information for the corresponding backend in the `config.yaml` file at the project root:

```yaml
# config.yaml
volcengine:
  access_key: "YOUR_VOLCENGINE_AK" # embedding still requires Volcengine ak/sk
  secret_key: "YOUR_VOLCENGINE_SK"

database:
  opensearch:
    host: "your_opensearch_host"
    port: 9200
    username: "your_username"
    password: "your_password"
```

**Step 2: Initialize the KnowledgeBase in your Python script**

In code, you only need to specify the `backend` name.

**Step 3: Add knowledge**

The method for adding knowledge is identical for both backends.

**Step 4: Integrate with the Agent**

```python
from veadk import Agent, Runner

agent = Agent(
    name="my_knowledgeable_agent",
    model_name="doubao-seed-1-6-250615",
    instruction="You are a knowledgeable assistant. Please use the knowledge base to answer questions.",
    knowledgebase=kb, # pass it here
    # ... other parameters
)

runner = Runner(agent, app_name="your_app_name")

# You can now run the runner normally
# ...
```

def calculate_date_difference(date1: str, date2: str) -> int:
    """
    Calculate the number of days between two dates.
    Args:
        date1: the first date, in "YYYY-MM-DD" format
        date2: the second date, in "YYYY-MM-DD" format
    Returns:
        the number of days between the two dates (absolute value)
    """
    # Parse the date strings into datetime objects
    try:
        d1 = datetime.strptime(date1, "%Y-%m-%d")
        d2 = datetime.strptime(date2, "%Y-%m-%d")
    except ValueError as e:
        raise ValueError(f"Invalid date format, please use YYYY-MM-DD: {e}")
    # Compute the difference and return the absolute value
    delta = d2 - d1
    return abs(delta.days)

type CalculateDateDifferenceArgs struct {
	Date1 string `json:"date1" jsonschema:"第一个日期，格式为YYYY-MM-DD"`
	Date2 string `json:"date2" jsonschema:"第二个日期，格式为YYYY-MM-DD"`
}

func CalculateDateDifferenceTool() (tool.Tool, error) {
	handler := func(ctx tool.Context, args CalculateDateDifferenceArgs) (map[string]any, error) {
		diff, err := CalculateDateDifference(args.Date1, args.Date2)
		if err != nil {
			return nil, err
		}
		return map[string]any{"result": diff}, nil
	}
	return functiontool.New(
		functiontool.Config{
			Name:        "calculate date difference",
			Description: "计算两个日期之间的天数差异\nArgs:\n\tdate1: 第一个日期，格式为YYYY-MM-DD\n\tdate2: 第二个日期，格式为YYYY-MM-DD\nReturns:\n\t两个日期之间的天数差异（绝对值）",
		},
		handler,
	)
}

func CalculateDateDifference(date1 string, date2 string) (int, error) {
	d1, err := time.Parse("2006-01-02", strings.TrimSpace(date1))
	if err != nil {
		return 0, fmt.Errorf("日期格式错误，请使用YYYY-MM-DD格式: %v", err)
	}
	d2, err := time.Parse("2006-01-02", strings.TrimSpace(date2))
	if err != nil {
		return 0, fmt.Errorf("日期格式错误，请使用YYYY-MM-DD格式: %v", err)
	}
	delta := d2.Sub(d1)
	days := int(math.Abs(delta.Hours() / 24))
	return days, nil
}

def web_search(query: str) -> str:
    """
    Use this tool to search the internet for real-time or external information when the knowledge base is insufficient. The query can be a single string or a JSON-formatted list of strings.
    """
    # This is a placeholder for the actual web search tool call
    # In a real scenario, this would be implemented by the environment.
    print(f"Performing web search for: {query}")

    # The model might send a list of queries directly, or a JSON string of a list.
    if isinstance(query, list):
        return f"Web search results for queries: {', '.join(query)}"

    # If it's a string, it could be a JSON list.
    if isinstance(query, str):
        try:
            queries = json.loads(query)
            if isinstance(queries, list):
                return f"Web search results for queries: {', '.join(queries)}"
        except (json.JSONDecodeError, TypeError):
            # It's just a plain string.
            pass
        # Fallthrough for plain string
        return "Web search results for '" + query + "'"

    # Fallback for any other unexpected type
    return f"Cannot process web search for unexpected query type: {query}"

def calculate_date_difference(date1: str, date2: str) -> int:
    """
    Calculate the number of days between two dates.
    Args:
        date1: the first date, in "YYYY-MM-DD" format
        date2: the second date, in "YYYY-MM-DD" format
    Returns:
        the number of days between the two dates (absolute value)
    """
    # Parse the date strings into datetime objects
    try:
        d1 = datetime.strptime(date1, "%Y-%m-%d")
        d2 = datetime.strptime(date2, "%Y-%m-%d")
    except ValueError as e:
        raise ValueError(f"Invalid date format, please use YYYY-MM-DD: {e}")
    # Compute the difference and return the absolute value
    delta = d2 - d1
    return abs(delta.days)

## Summary and Comparison

In short, the KnowledgeBase design lets you easily switch the underlying vector database without changing most of your business logic. The main differences lie in the upfront configuration and the fact that viking requires explicitly creating a collection.

| Feature | Viking Backend | OpenSearch Backend |
| :--- | :--- | :--- |
| **Core advantage** | Managed service, no ops overhead, tightly integrated with the Volcengine ecosystem. | Open-source and flexible, supports private deployment, with a mature community ecosystem. |
| **Configuration (`config.yaml`)** | Requires `viking.project` and `viking.region`. | Requires `opensearch.host`, `port`, `username`, `password`. |
| **Initialization** | Requires manually checking and calling `kb.create_collection()` before use. | Usually creates the index automatically, with no extra steps. |
| **Code usage** | **Identical**. Methods like `add_from_files` and `add_from_text` are used the same way. | **Identical**. The `veadk` framework's abstraction layer handles this well. |
| **Dependencies** | Relies on Volcengine `ak/sk` for authentication and embedding. | Also requires `ak/sk` to call the embedding model, but the database itself is independent. |
